## 2. Naming Conventions

### Files
- All file names are `snake_case`.
- Bloc files: `<feature>_bloc.dart`, `<feature>_event.dart`, `<feature>_state.dart`
- Repository files: `<feature>_repository.dart`
- Service files: `<feature>_service.dart`
- Model files: `<feature>.dart` (singular noun, e.g., `user.dart`, `order.dart`)
- Entity/Params files: `<feature>_params.dart`
- Screen files: `<feature>.dart` or `<screen_name>.dart`
- Widget files: `<widget_name>.dart` (descriptive noun, e.g., `item_card.dart`, `loading.dart`)

### Classes
- Bloc: `<Feature>Bloc` — e.g., `SendOTPBloc`, `OrderBloc`
- Event base: `<Feature>Event` — e.g., `SendOTPEvent`
- State base: `<Feature>State` — e.g., `SendOTPState`
- Repository abstract: `<Feature>Repository` — e.g., `AuthRepository`
- Repository impl: `<Feature>RepositoryImpl` — e.g., `AuthRepositoryImpl`
- Service: `<Feature>Service` — e.g., `AuthService`
- Model: PascalCase singular — e.g., `User`, `Order`, `JsonResponse`
- Params/Entity: `<Feature>Params` or `<Feature>ParamsItem` — e.g., `OrderParams`
- Screen widgets: `<Feature>Page` or `<Feature>Screen` — e.g., `AuthPage`
- Reusable widgets: descriptive PascalCase — e.g., `ItemCard`, `Loading`

### Constants
- All top-level constants use the `k` prefix in camelCase: `kURL`, `kSendOTPPath`, `kAccessToken`
- Route name strings use the suffix `Route`: `authRoute`, `verifyOTPRoute`

### Private methods in Bloc
- Event handler methods are prefixed with `_on`: `_onSendOTPEvent`, `_onVerifyOTPEvent`

## 3. BLoC Architecture — The Non-Negotiable Rules

### 3.1 File Structure — Always Use `part` / `part of`

Every bloc feature has **exactly 3 files**. The event and state files are `part of` the bloc file.

```dart
// send_otp_bloc.dart
part 'send_otp_event.dart';
part 'send_otp_state.dart';
```

```dart
// send_otp_event.dart
part of 'send_otp_bloc.dart';
```

```dart
// send_otp_state.dart
part of 'send_otp_bloc.dart';
```

**NEVER make event or state files standalone — they must always be `part of` the bloc file.**

---

### 3.2 State Pattern — TWO Styles, Pick the Right One

There are **two state patterns** in use. Choose based on the complexity and nature of the bloc.

---

#### 🔵 Style A — Single State Class with `isLoading` / `error` / `message` + `copyWith()`

> **Use this for blocs that:**
> - Hold persistent data across multiple events (e.g., a list, the current user, a cart)
> - Have multiple events that each update different parts of the same state
> - Need to be persisted with `HydratedBloc`
> - Update state granularly without replacing the whole state

```dart
part of '<feature>_bloc.dart';

/// [<Feature>State] holds all state for the <feature> feature.
/// [isLoading] is true when any operation is in progress.
/// [error] is non-empty when an error has occurred.
/// [message] is non-empty when a success message should be shown.
/// [data] holds the primary data for this feature.
class <Feature>State extends Equatable {
  const <Feature>State({
    this.isLoading = false,
    this.error = '',
    this.message = '',
    this.data = const <SomeModel>[],    // ← adapt fields to the feature
  });

  final bool isLoading;
  final String error;
  final String message;
  final List<SomeModel> data;

  /// [copyWith] returns a new [<Feature>State] with only the given fields replaced.
  /// All unchanged fields retain their current values.
  <Feature>State copyWith({
    bool? isLoading,
    String? error,
    String? message,
    List<SomeModel>? data,
  }) =>
      <Feature>State(
        isLoading: isLoading ?? this.isLoading,
        error: error ?? this.error,
        message: message ?? this.message,
        data: data ?? this.data,
      );

  @override
  List<Object?> get props => [isLoading, error, message, data];
}
```

**When `HydratedBloc` is used**, also add `fromJson` and `toJson`:

```dart
/// [fromJson] reconstructs state from persisted storage.
/// Only persist the fields that must survive app restarts.
/// Prefix hydrated keys with 'Hydrated' to distinguish them.
factory <Feature>State.fromJson(Map<String, dynamic> json) => <Feature>State(
      data: (json['HydratedData'] as List?)
              ?.map<SomeModel>((e) => SomeModel.fromJson(e))
              .toList() ??
          const <SomeModel>[],
    );

/// [toJson] serializes only the fields that need to be persisted.
Map<String, dynamic> toJson() => {
      'HydratedData': data.map((e) => e.toJson()).toList(),
    };
```

**How to emit with Style A (in the bloc):**

```dart
// Start of any async event handler — clear error/message, show loading
emit(state.copyWith(isLoading: true, error: '', message: ''));

// On success — turn off loading, set message, update data
emit(state.copyWith(
  isLoading: false,
  message: 'Loaded successfully.',
  data: result,
));

// On failure — turn off loading, set error
emit(state.copyWith(isLoading: false, error: response.message));

// On exception catch
emit(state.copyWith(isLoading: false, error: 'Something went wrong'));

// Reset to empty state entirely
emit(const <Feature>State());
```

**UI side with Style A:**

```dart
BlocBuilder<<Feature>Bloc, <Feature>State>(
  builder: (context, state) {
    if (state.isLoading) return const Loading();
    if (state.error.isNotEmpty) { /* show error */ }
    return YourWidget(data: state.data);
  },
)

// For side effects (snackbar, navigation) triggered by message/error changes:
BlocListener<<Feature>Bloc, <Feature>State>(
  listenWhen: (prev, curr) => prev.message != curr.message && curr.message.isNotEmpty,
  listener: (context, state) {
    ScaffoldMessenger.of(context).showSnackBar(
      SnackBar(content: Text(state.message)),
    );
  },
)
```

#### 🟢 Style B — Multiple Subclasses (Abstract base + named subclasses)

> **Use this for blocs that:**
> - Perform a **single, focused action** with a clear start/end (e.g., send OTP, update profile)
> - Do NOT need to hold or accumulate data across events
> - Are simple enough that the state can only ever be one thing at a time

```dart
part of '<feature>_bloc.dart';

/// [<Feature>State] is the abstract base class for all states.
abstract class <Feature>State extends Equatable {
  const <Feature>State();

  @override
  List<Object> get props => [];
}

/// [<Feature>Initial] is the initial/idle state.
class <Feature>Initial extends <Feature>State {}

/// [<Feature>Loading] is emitted immediately when the event fires.
/// Use descriptive present-progressive naming: SendingOTP, VerifyingOTP.
class <Feature>Loading extends <Feature>State {}

/// [<Feature>Success] is emitted when the operation completes successfully.
class <Feature>Success extends <Feature>State {
  const <Feature>Success();
}

/// [<Feature>Failure] is emitted when the operation fails.
/// [message] contains the human-readable error message — always required.
class <Feature>Failure extends <Feature>State {
  const <Feature>Failure({required this.message});
  final String message;

  @override
  List<Object> get props => [message];
}
```

**Rules for Style B:**
- The abstract base always has a `const <Feature>State()` constructor and returns empty `props`.
- `<Feature>Loading` carries no data — it is a pure signal to show a spinner.
- `<Feature>Success` can optionally carry data fields. If it does, override `props` with those fields.
- `<Feature>Failure` **always** has a `required String message` field — never emit a failure without a message.
- States with no data do not override `props`.
- Use `const` constructors everywhere possible.

**How to emit with Style B (in the bloc):**

```dart
emit(<Feature>Loading()); // or emit(SendingOTP())
try {
  final response = await repository.someMethod(event.param);
  if (response.success) {
    emit(const <Feature>Success());
  } else {
    emit(<Feature>Failure(message: response.message));
  }
} catch (_) {
  emit(const <Feature>Failure(message: 'Something went wrong'));
}
```

**UI side with Style B:**

```dart
BlocConsumer<<Feature>Bloc, <Feature>State>(
  listener: (context, state) {
    if (state is <Feature>Success) {
      // navigate or show success feedback
    } else if (state is <Feature>Failure) {
      ScaffoldMessenger.of(context).showSnackBar(
        SnackBar(
          content: Text(state.message),
          backgroundColor: Theme.of(context).colorScheme.error,
        ),
      );
    }
  },
  builder: (context, state) {
    if (state is <Feature>Loading) return const Loading();
    return YourNormalUI();
  },
)
```

---

#### ✅ Quick Decision Table

| Situation | Use Style |
|---|---|
| Bloc holds a list/object that persists and updates across many events | **A** |
| Bloc uses `HydratedBloc` for persistence | **A** |
| Multiple events update different slices of the same state | **A** |
| Bloc does one simple async action (send, verify, update) | **B** |
| State only ever means one thing at a time (loading OR success OR failure) | **B** |
| Bloc needs `copyWith()` to partially update state | **A** |

### 3.3 Event Pattern

```dart
part of '<feature>_bloc.dart';

/// [<Feature>Event] is the event description.
/// [param] is the description of the parameter.
class <Feature>Event extends Equatable {
  const <Feature>Event({required this.param});
  final String param;

  @override
  List<Object> get props => [param];
}
```

**Rules:**
- All events extend `Equatable`.
- All event parameters must be `final`.
- Use named parameters with `required` for all event fields.
- Override `props` with all fields.
- If an event has no data, it still extends `Equatable` with empty `props`.
- Events that take no parameters can be `const` and passed without `()` when adding.

### 3.4 Bloc Class Pattern

The bloc class structure mirrors the state style chosen in 3.2.

---

#### Style A Bloc — Single State with `copyWith()`

```dart
import 'dart:async';

import 'package:<app>/data/repositories/<feature>_repository.dart';
import 'package:<app>/data/models/<feature_model>.dart';
import 'package:equatable/equatable.dart';
import 'package:hydrated_bloc/hydrated_bloc.dart'; // or flutter_bloc if not hydrated

part '<feature>_event.dart';
part '<feature>_state.dart';

/// [<Feature>Bloc] manages the state of <feature>.
/// Uses a single state class updated via [copyWith].
/// [isLoading], [error], and [message] in state signal async operation status.
///
/// Events:
/// 1) [Load<Feature>Event] — fetches data from the server.
/// 2) [Refresh<Feature>Event] — resets and reloads data.
class <Feature>Bloc extends HydratedBloc<<Feature>Event, <Feature>State> {
  <Feature>Bloc({required this.repository}) : super(const <Feature>State()) {
    on<Load<Feature>Event>(_onLoad<Feature>Event);
  }

  final <Feature>Repository repository;

  FutureOr<void> _onLoad<Feature>Event(
      Load<Feature>Event event, Emitter<<Feature>State> emit) async {
    try {
      emit(state.copyWith(isLoading: true, error: '', message: ''));
      final response = await repository.getData();
      if (response.success) {
        final data = response.data as List<<Feature>Model>;
        emit(state.copyWith(
          isLoading: false,
          data: data,
          message: 'Loaded successfully.',
        ));
      } else {
        emit(state.copyWith(isLoading: false, error: response.message));
      }
    } on Exception catch (e) {
      emit(state.copyWith(isLoading: false, error: e.toString()));
    }
  }

  @override
  <Feature>State? fromJson(Map<String, dynamic> json) =>
      <Feature>State.fromJson(json);

  @override
  Map<String, dynamic>? toJson(<Feature>State state) => state.toJson();
}
```

#### Style B Bloc — Multiple State Subclasses

```dart
import 'dart:async';

import 'package:<app>/data/repositories/<feature>_repository.dart';
import 'package:equatable/equatable.dart';
import 'package:flutter_bloc/flutter_bloc.dart';

part '<feature>_event.dart';
part '<feature>_state.dart';

/// [<Feature>Bloc] manages the state of <feature>.
/// [<Feature>Bloc] extends [Bloc] which takes [<Feature>Event] and returns [<Feature>State].
/// [repository] is the object of [<Feature>Repository].
///
/// States:
/// 1) [<Feature>Initial] — initial idle state.
/// 2) [<Feature>Loading] — operation in progress.
/// 3) [<Feature>Success] — operation completed successfully.
/// 4) [<Feature>Failure] — operation failed with a [message].
///
/// Events:
/// 1) [<Feature>Event] — triggers the operation.
///
/// [_on<Feature>Event] — calls the repository and emits the appropriate state.
class <Feature>Bloc extends Bloc<<Feature>Event, <Feature>State> {
  <Feature>Bloc({required this.repository}) : super(<Feature>Initial()) {
    on<<Feature>Event>(_on<Feature>Event);
  }

  final <Feature>Repository repository;

  FutureOr<void> _on<Feature>Event(
      <Feature>Event event, Emitter<<Feature>State> emit) async {
    emit(<Feature>Loading());
    try {
      final response = await repository.someMethod(event.param);
      if (response.success) {
        emit(const <Feature>Success());
      } else {
        emit(<Feature>Failure(message: response.message));
      }
    } catch (_) {
      emit(const <Feature>Failure(message: 'Something went wrong'));
    }
  }
}
```

**Rules (both styles):**
- Always emit the loading state as the **very first** `emit` inside every async event handler — before `await`.
- Wrap all async logic in `try/catch`.
- On `DioException`, the service layer already converts it to `JsonResponse.failure(...)` — so the bloc's `catch` is for truly unexpected exceptions only.
- Repository is **always** injected via constructor — never instantiate inside the bloc.
- Dependent blocs (e.g., `UserBloc`) are injected via constructor and declared as `final` fields.
- The `FutureOr<void>` return type is mandatory for all event handlers.
- Event handlers are always private: `_on<Feature>Event`.
- Declare `final` fields (repository, injected blocs) **after** the constructor, not before it.

### 3.5 Local Storage Rules

The project may use **one or more** of these storage strategies depending on what packages are installed. Check `pubspec.yaml` first to confirm which packages are present, then follow the corresponding pattern below.

---

Preferred storage strategies:

- `hydrated_bloc` → persistent bloc state
- `flutter_secure_storage` → sensitive small values (tokens, IDs)
- `hive` / `isar` → large structured local caching

Rules:

- Initialize storage before `runApp()`
- Never access storage directly from UI
- Use dedicated services/repositories for storage operations
- Persist only necessary data
- Avoid storing large objects in secure storage
- Cache failures must never crash the app


### 3.6 Inter-Bloc Communication

When one bloc depends on another (e.g., `VerifyOTPBloc` depends on `UserBloc`):

```dart
class VerifyOTPBloc extends Bloc<VerifyOTPEvent, VerifyOTPState> {
  VerifyOTPBloc({
    required this.authRepository,
    required this.userBloc,
  }) : super(VerifyOTPInitial()) {
    on<VerifyOTPEvent>(_onVerifyOTPEvent);
  }

  FutureOr<void> _onVerifyOTPEvent(
      VerifyOTPEvent event, Emitter<VerifyOTPState> emit) async {
    emit(VerifyingOTP());
    try {
      final response = await authRepository.verifyOTP(event.phone, event.otp);
      if (response.success) {
        final user = response.data as User;
        userBloc.add(SaveUser(user: user));            // ← cross-bloc event dispatch
        await HydratedBloc.storage.write(kAccessToken, user.accessToken);
        emit(const VerifyOTPSuccess());
      } else {
        emit(VerifyOTPFailure(message: response.message));
      }
    } on Exception catch (e) {
      emit(VerifyOTPFailure(message: 'Error: $e'));
    }
  }

  final AuthRepository authRepository;
  final UserBloc userBloc;
}
```

**Rules:**
- Only call `otherBloc.add(SomeEvent(...))` — **never** access another bloc's state directly inside a bloc.
- Dependent blocs are always injected via constructor.
- Declare injected blocs as `final` fields.

> ```yaml
> flutter:
>   assets:
>     - .env
> ```

**Reading values from `.env`:**
```dart
import 'package:flutter_dotenv/flutter_dotenv.dart';

// Always use dotenv.get() with a fallback — never use dotenv['KEY'] directly
final baseUrl = dotenv.get('BASE_URL', fallback: '');
final httpScheme = dotenv.get('HTTP_SCHEME', fallback: 'https');
final apiPath = dotenv.get('API_PATH', fallback: '/api');
final apiVersion = dotenv.get('API_VERSION', fallback: '/v1');

// Compose the full URL:
final kURL = '$httpScheme://$baseUrl$apiPath$apiVersion';
```

```dart
// In a service constructor or a constants file:
final kURL = '${dotenv.get('HTTP_SCHEME')}://${dotenv.get('BASE_URL')}${dotenv.get('API_PATH')}${dotenv.get('API_VERSION')}';
```

**Rules:**
- **Never** commit the real production `.env` file to version control — add `.env` to `.gitignore`.
- Always provide a `.env.example` file with placeholder values for onboarding.
- Use `dotenv.get('KEY', fallback: '')` — always specify a fallback, never assume the key exists.
- Environment-specific configs (dev/staging/prod) use separate `.env.development`, `.env.production` files and are loaded conditionally in `start.dart`.

## 4. Repository Pattern

### 4.1 Abstract + Implementation

Every repository has an **abstract class** and a **concrete Impl class** in the same file:

```dart
import 'package:<app>/data/models/json_response.dart';
import 'package:<app>/data/services/<feature>_service.dart';

/// [<Feature>Repository] defines the contract for <feature> operations.
abstract class <Feature>Repository {
  /// [methodName] description of what it does.
  Future<JsonResponse> methodName({
    required String param1,
    int param2 = 0,
  });
}

class <Feature>RepositoryImpl implements <Feature>Repository {
  const <Feature>RepositoryImpl({required this.<feature>Service});
  final <Feature>Service <feature>Service;

  @override
  Future<JsonResponse> methodName({
    required String param1,
    int param2 = 0,
  }) =>
      <feature>Service.methodName(param1: param1, param2: param2);
}
```

**Rules:**
- The abstract class documents every method with `///` doc comments.
- The `Impl` class uses `const` constructor.
- The `Impl` class simply delegates to the service — **no business logic** inside the impl.
- The `Impl` class always uses arrow (`=>`) syntax for single-expression overrides.
- Method signatures in the abstract and impl must match exactly, including optional/named parameters and defaults.
- Return type is always `Future<JsonResponse>` (from `json_response.dart`).
- Inject the service via required named parameter in the constructor.

## 5. Service Pattern (Dio HTTP Layer)

### 5.1 Service Class Structure

```dart
import 'package:dio/dio.dart';
import 'package:<app>/core/exceptions/dio_exception.dart';
import 'package:<app>/core/interceptors/auth_interceptor.dart';
import 'package:<app>/core/interceptors/x_client_interceptor.dart';
import 'package:<app>/core/utils/constants.dart';
import 'package:<app>/data/models/json_response.dart';

/// [<Feature>Service] handles all <feature>-related HTTP operations.
class <Feature>Service {
  <Feature>Service({required this.dio}) {
    dio.options.baseUrl = '$kURL$k<Feature>Path';
    dio.options.sendTimeout = const Duration(seconds: 10);
    dio.options.receiveTimeout = const Duration(seconds: 10);
    dio.options.connectTimeout = const Duration(seconds: 10);
    dio.options.responseType = ResponseType.json;
    dio.options.headers = {
      'Content-Type': 'application/json',
      'Accept': 'application/json',
    };
    dio.interceptors.add(XClientInterceptor());
    dio.interceptors.add(AuthInterceptor());  // add only for authenticated routes
  }

  final Dio dio;

  Future<JsonResponse> methodName({required String param}) async {
    try {
      final response = await dio.get(kSomePath, queryParameters: {'param': param});
      if (response.statusCode == 200) {
        final model = SomeModel.fromJson(response.data);
        return JsonResponse.success(
          message: response.data['message'] ?? 'Success',
          data: model,
        );
      } else {
        return JsonResponse.failure(
          statusCode: response.statusCode ?? 500,
          message: response.data['message'] ?? 'Something went wrong!',
        );
      }
    } on DioException catch (error) {
      final message = DioExceptionOf.exceptionFromDioError(error).errorMessage;
      return JsonResponse.failure(
        statusCode: error.response?.statusCode ?? 500,
        message: 'Error: $message',
      );
    } on Exception {
      return JsonResponse.failure(
        statusCode: 500,
        message: 'Error: Unexpected error occurred.',
      );
    }
  }
}
```

**Rules:**
- Dio is always injected via constructor — **never** instantiate `Dio()` inside a service.
- Configure `baseUrl`, timeouts, headers, and interceptors **inside the constructor body** — not in a separate `init` method.
- Always set `sendTimeout`, `receiveTimeout`, and `connectTimeout` to `Duration(seconds: 10)`.
- Always set `responseType` to `ResponseType.json`.
- Always set `Content-Type` and `Accept` headers to `application/json`.
- Always add `XClientInterceptor()` to every service.
- Add `AuthInterceptor()` only for services that require authentication.
- **All HTTP methods return `Future<JsonResponse>`** — never return raw `Response` or model types from a service.
- Always handle 3 layers of exceptions: success path, `on DioException`, and `on Exception`.
- Parse models inside the success branch of the service — pass the parsed model as `data` in `JsonResponse.success(data: model)`.
- Use `response.data['message']` with a null-safe fallback string for all message fields.

## 6. Model Pattern

### 6.1 Response Models (Immutable — `data/models/`)

```dart
import 'package:equatable/equatable.dart';

/// [ModelName] represents a <description>.
class ModelName extends Equatable {
  const ModelName({
    required this.id,
    this.name,
    this.createdAt,
  });

  final String id;
  final String? name;
  final DateTime? createdAt;

  factory ModelName.fromJson(Map<String, dynamic> json) => ModelName(
        id: json['_id']?.toString() ?? '',
        name: json['name']?.toString(),
        createdAt: DateTime.tryParse('${json['createdAt']}'),
      );

  Map<String, dynamic> toJson() => {
        '_id': id,
        'name': name,
        'createdAt': createdAt?.toIso8601String(),
      };

  ModelName copyWith({
    String? id,
    String? name,
    DateTime? createdAt,
  }) =>
      ModelName(
        id: id ?? this.id,
        name: name ?? this.name,
        createdAt: createdAt ?? this.createdAt,
      );

  @override
  List<Object?> get props => [id, name, createdAt];
}
```

**Rules:**
- All models in `data/models/` extend `Equatable`.
- All fields are `final` (immutable).
- All models have: `const` constructor, `factory fromJson`, `toJson()`, `copyWith()`, and `get props`.
- Use `?.toString()` for safe string parsing from JSON — never cast directly.
- Use `DateTime.tryParse('${json['field']}')` for dates — never `DateTime.parse()`.
- Use `int.tryParse`, `double.tryParse`, `bool.tryParse`, `num.tryParse` for numeric/bool fields — never direct cast.
- Include **all** fields in `props`.
- For nested models, call `NestedModel.fromJson(json['nestedField'])`.
- For list fields: `(json['items'] as List?)?.map((e) => Item.fromJson(e)).toList() ?? []`.

### 6.2 `JsonResponse` — Universal Wrapper

This model is **pre-built and must never be modified**. It is the **only return type** for service methods.

```dart
// Usage in services:
return JsonResponse.success(message: 'Done', data: parsedModel);
return JsonResponse.failure(statusCode: 404, message: 'Not found');
return JsonResponse.created(message: 'Created', data: parsedModel);

// Usage in bloc:
if (response.success) {
  final model = response.data as ModelName;
  emit(<Feature>Success(data: model));
} else {
  emit(<Feature>Failure(message: response.message));
}
```

## 7. Entity / Params Pattern (`data/entity/`)

Params classes are used for **request bodies**. They are **mutable** and do **not** extend `Equatable`:

```dart
class <Feature>Params {
  <Feature>Params({
    required this.field1,
    required this.field2,
    this.optionalField,
  });

  String field1;
  String field2;
  String? optionalField;

  Map<String, dynamic> toJson() => {
        'field1': field1,
        'field2': field2,
        'optionalField': optionalField,
      };

  <Feature>Params copyWith({
    String? field1,
    String? field2,
    String? optionalField,
  }) {
    return <Feature>Params(
      field1: field1 ?? this.field1,
      field2: field2 ?? this.field2,
      optionalField: optionalField ?? this.optionalField,
    );
  }
}
```

**Rules:**
- Entity/Params classes live in `data/entity/` — **not** in `data/models/`.
- They are mutable — fields are **not** `final`.
- They do **not** extend `Equatable`.
- They always have `toJson()` and `copyWith()`.
- They do **not** have `fromJson()` — they are outbound-only (request bodies).

## 8. UI / Screen Pattern

### 8.1 Screen Widget Structure

```dart
class <Feature>Page extends StatefulWidget {
  const <Feature>Page({super.key});

  @override
  State<<Feature>Page> createState() => _<Feature>PageState();
}

class _<Feature>PageState extends State<<Feature>Page> {
  // Controllers and local state declared here
  final _controller = TextEditingController();

  @override
  void dispose() {
    _controller.dispose();
    super.dispose();
  }

  @override
  Widget build(BuildContext context) {
    return Scaffold(
      body: BlocConsumer<<Feature>Bloc, <Feature>State>(
        listener: (context, state) {
          if (state is <Feature>Success) {
            // navigation, snackbar, etc.
          } else if (state is <Feature>Failure) {
            ScaffoldMessenger.of(context).showSnackBar(
              SnackBar(
                content: Text(state.message),
                backgroundColor: Theme.of(context).colorScheme.error,
              ),
            );
          }
        },
        builder: (context, state) {
          if (state is <Feature>Loading) {
            return const Loading();
          }
          return // normal UI
        },
      ),
    );
  }
}
```

### 8.2 `BlocConsumer` vs `BlocBuilder` vs `BlocListener`

| Widget | When to use |
|---|---|
| `BlocConsumer` | When you need **both** side effects (navigation, snackbar) **and** UI rebuilds |
| `BlocBuilder` | When you need **only** UI rebuilds, no side effects |
| `BlocListener` | When you need **only** side effects, no UI rebuild |

**Rules:**
- Use `BlocConsumer` as the default for screens that make API calls (the listener handles side effects, the builder handles UI).
- Use `buildWhen` on `BlocBuilder` when rebuilds should be selective to avoid unnecessary redraws.
- Always handle `<Feature>Loading` state in the builder — show a loading widget, never leave it unhandled.
- Always handle `<Feature>Failure` state in the listener — show a `SnackBar` with `state.message`.
- Access blocs with `context.read<<Feature>Bloc>()` when **dispatching events** (inside callbacks).
- Access blocs with `BlocProvider.of<<Feature>Bloc>(context)` when **reading state** outside a builder.
- **Never** call `context.read<>()` or `BlocProvider.of<>()` inside a `build()` method for state reading — use `BlocBuilder` or `BlocConsumer` for that.

### 8.3 Dispatching Events from UI

```dart
// Correct — inside a button callback or gesture:
context.read<SendOTPBloc>().add(
  SendOTPEvent(phone: phone.text.trim()),
);

// Correct — on init:
context.read<SomeBloc>()..add(const LoadEvent());
```

### 8.4 Theming — Always Use Theme Context

```dart
// ✅ Correct
Theme.of(context).colorScheme.primary
Theme.of(context).textTheme.bodyLarge
Theme.of(context).colorScheme.error
Theme.of(context).colorScheme.surface

// ❌ Wrong
Colors.blue
Colors.red
const TextStyle(fontSize: 16, color: Colors.black)
```

**Rules:**
- Never use hardcoded `Colors.*` in screens or widgets.
- Always derive colors from `Theme.of(context).colorScheme.*`.
- Always derive text styles from `Theme.of(context).textTheme.*`, then use `.copyWith()` if modification is needed.
- Use `MediaQuery.of(context).size` or the `responsive.dart` helpers (`compact()`, `medium()`, `responsivePadding()`) for responsive layout.

## 9. App Bootstrap & `MultiBlocProvider` Setup

### 9.1 `app/start.dart` — Entry Point

`startApplication()` is always called from `main.dart`:

```dart
// main.dart
void main() => startApplication(() => const Application());
```

This handles:
1. `WidgetsFlutterBinding.ensureInitialized()`
2. Firebase initialization
3. HydratedBloc encrypted storage setup
4. Bloc observer setup
5. Orientation lock
6. URL strategy (web)
7. `runApp()`

### 9.2 `app/app.dart` — `MultiBlocProvider`

All blocs are registered in `MultiBlocProvider` in `app.dart`. The pattern for registering a bloc:

```dart
BlocProvider<<Feature>Bloc>(
  create: (context) => <Feature>Bloc(
    repository: <Feature>RepositoryImpl(
      <feature>Service: <Feature>Service(
        dio: Dio(),
      ),
    ),
  ),
),
```

**Rules:**
- Every `Dio()` instance is created **fresh** per service inside the `BlocProvider` — never share a single global `Dio` instance.
- Each service gets its **own** `Dio()` — services configure their own base URL and interceptors in their constructor.
- If a bloc depends on another bloc, use `BlocProvider.of<OtherBloc>(context)` inside the `create` callback:

```dart
BlocProvider<DependentBloc>(
  create: (context) => DependentBloc(
    otherBloc: BlocProvider.of<OtherBloc>(context),
    repository: ...,
  ),
),
```

- The order of `BlocProvider` registration matters — a bloc must be registered **before** any other bloc that depends on it.
- Use `lazy: false` only when a bloc must initialize immediately on app start (e.g., `GSTMasterBloc`, `DeviceBloc`).
- Dispatch initialization events using the cascade operator: `..add(const InitEvent())`.

## 10. Routing — GoRouter

> **Every time a new screen is added, BOTH `routes.dart` and `app_router.dart` MUST be updated.** This is mandatory — no screen is reachable without a registered route.

### 10.1 `routes.dart` — Route Name Constants

Every route name is a `const String` in `lib/core/route/routes.dart`. Add a new one for every new screen:

```dart
// routes.dart
const String authRoute = 'auth';
const String homeRoute = 'home';
const String verifyOTPRoute = 'verify-otp';
const String <newFeature>Route = '<new-feature>';  // ← add this for every new screen
```

**Rules:**
- Route name values are `kebab-case` strings (lowercase with hyphens).
- Constant names are `camelCase` with `Route` suffix.
- Never use a hardcoded string for navigation — always reference the constant.

---

### 10.2 `app_router.dart` — GoRoute Registration

Every route constant must have a matching `GoRoute` inside `app_router.dart`. Add new routes inside the existing `routes: [...]` list:

```dart
// Inside app_router.dart routes list:
GoRoute(
  name: <newFeature>Route,
  path: '/<new-feature>',
  builder: (context, state) => const <NewFeature>Page(),
),

// With query parameters:
GoRoute(
  name: verifyOTPRoute,
  path: '/verify-otp',
  builder: (context, state) {
    final phone = state.uri.queryParameters['phone'] ?? '';
    return VerifyOTPPage(phone: phone);
  },
),

// Nested routes (child of parent):
GoRoute(
  name: parentRoute,
  path: '/parent',
  builder: (context, state) => const ParentPage(),
  routes: [
    GoRoute(
      name: childRoute,
      path: 'child',   // ← no leading slash for nested routes
      builder: (context, state) => const ChildPage(),
    ),
  ],
),
```

**Rules:**
- Path values start with `/` for top-level routes, **no** leading slash for nested routes.
- Always use `name:` parameter — never navigate by path string.
- Route guards (`redirect`) belong in `app_router.dart`, not in individual screens.
- Shell routes (bottom nav, drawer) use `ShellRoute` — wrap child routes inside.
- Keep imports in `app_router.dart` alphabetically sorted by feature name.

---

### 10.3 Navigation API — Always Use Named Routes

```dart
// Push a new screen onto the stack:
context.pushNamed(verifyOTPRoute);

// Replace current screen (no back button):
context.pushReplacementNamed(homeRoute);

// Go to a screen and clear the entire stack:
context.goNamed(homeRoute);

// With query parameters:
context.pushNamed(
  verifyOTPRoute,
  queryParameters: {'phone': phone.text.trim()},
);

// With path parameters:
context.pushNamed(
  orderDetailRoute,
  pathParameters: {'id': orderId},
);

// Pop back:
context.pop();

// Pop with a result value:
context.pop(result);
```

**Never use:**
- `Navigator.of(context).push(...)` or `Navigator.push(...)`
- Raw path strings: `context.go('/home')`
- `MaterialPageRoute` directly in screens

## 12. Error Handling Rules

1. **Service layer**: Always wrap Dio calls in `try/catch` with 3 layers:
   - Success path (`statusCode == 200` check)
   - `on DioException catch (error)` — use `DioExceptionOf.exceptionFromDioError(error).errorMessage`
   - `on Exception` — generic fallback

2. **Bloc layer**: Wrap the `await repository.method()` in `try/catch (_)` — emit `<Feature>Failure` in catch.

3. **UI layer**: Always show errors via `SnackBar` with `Theme.of(context).colorScheme.error` background. Never use `AlertDialog` for transient API errors.

4. **Never** `print()` errors. Use the `AppBlocObserver` for bloc event/state logging (already configured — do not touch).

## 13. Equatable Rules

- **Models** (`data/models/`): Always extend `Equatable`. Override `get props` with ALL fields.
- **States**: Always extend `Equatable`. Override `get props` only in states that carry data.
- **Events**: Always extend `Equatable`. Override `get props` with all event parameters.
- **Params/Entity** (`data/entity/`): Do **NOT** extend `Equatable` — these are mutable request objects.
- Use `List<Object?>` (with nullable) when any field can be null. Use `List<Object>` when all fields are non-null.

## 15. What is Strictly Forbidden

- ❌ Using `setState` for any logic that is managed by BLoC — `setState` is allowed only for purely local UI state (e.g., form validation flag, animation controller).
- ❌ Calling `context.read<>()` or `BlocProvider.of<>()` at the top level of `build()` — only call inside callbacks or use `BlocBuilder`/`BlocConsumer`.
- ❌ Hardcoded colors (`Colors.red`, `Colors.blue`, etc.) in screens or widgets.
- ❌ Putting business logic inside repository `Impl` — the impl delegates directly to the service.
- ❌ Making HTTP calls directly from a bloc — all HTTP goes through service → repository → bloc.
- ❌ Sharing a single `Dio` instance across services.
- ❌ Throwing exceptions from repositories or blocs — catch them and emit failure states.
- ❌ Using raw `Navigator.push()` — always use GoRouter named routes.
- ❌ Modifying any file inside `core/interceptors/`, `core/utils/`, `core/exceptions/`, `core/environments/`, or `debug/`.
- ❌ Importing Firebase packages outside of `start.dart`, `firebase_options.dart`, and services that explicitly need them (e.g., `AuthService` for FCM token).
- ❌ Creating a bloc state that doesn't follow either Style A (single class with `isLoading`/`error`/`message`/`copyWith`) or Style B (abstract base + Initial/Loading/Success/Failure subclasses).
- ❌ Leaving `catch` blocks empty — always emit a failure state or log the error.
- ❌ Using `dynamic` as a variable type when the type is known.
- ❌ Prefer explicit typing when it improves readability or architectural clarity
- ❌ Use `var` only when the type is obvious from the right-hand side
- ❌ Avoid unnecessary `dynamic`

## 16. Quick Reference Checklist

When creating a new feature, follow this exact order:

- [ ] **Model** → `lib/data/models/<feature>.dart` (with `Equatable`, `fromJson`, `toJson`, `copyWith`)
- [ ] **Params** (if needed) → `lib/data/entity/<feature>_params.dart` (mutable, `toJson`, `copyWith`)
- [ ] **Service** → `lib/data/services/<feature>_service.dart` (Dio injected, constructor config, `JsonResponse` returns)
- [ ] **Repository** → `lib/data/repositories/<feature>_repository.dart` (abstract + impl)
- [ ] **BLoC** → `lib/bloc/<feature>/` (3 files: bloc, event, state — 4-state pattern)
- [ ] **Register BLoC** → `lib/app/app.dart` `MultiBlocProvider` list (in correct dependency order)

- [ ] **Route** → Add route name to `lib/core/route/routes.dart`, add `GoRoute` to `app_router.dart`
